1. Panoramica

"Messaggi di Sistema Pro" e' il sistema centralizzato dove l'admin gestisce TUTTI i messaggi (WhatsApp + email) di DR7. Vive nella tabella Supabase system_messages. Lo schema rilevante:

  • message_key: chiave univoca (es. pro_conferma_noleggio)
  • message_body: corpo template con placeholder {nome}, {vehicle_name}, ecc.
  • label: etichetta visibile nell'UI
  • is_enabled: attivo/spento
  • handled_events: array di event-key legacy gestiti
  • target_service_type: rental / car_wash / mechanical / prime_wash

Il render server-side passa per netlify/functions/utils/messageTemplates.ts (renderTemplate, getMessageTemplate) e il routing legacy→pro per netlify/functions/utils/proTemplateRouting.ts (OLD_TO_PRO map).

3. Findings seri — placeholder leak, inconsistenze

3.1 — pro_email_contratto: minimal vars set

File: send-contract-email.ts:102-107

Il sender passa solo 4 variabili: customer_name, nome, vehicle_name, booking_id. Se nel template (o nel subject) l'admin scrive {pickup_date}, {dropoff_date}, {plate}, {contract_number}, {total} — escono come testo letterale nel corpo email / nel subject.

3.2 — ReservationsTab passa pro_* direttamente (inconsistenza)

File: ReservationsTab.tsx:2992

Solo pro_annullamento_cliente passa la chiave canonical direttamente; tutti gli altri caller dello stesso file usano chiavi legacy (payment_link_customer, signature_request_link, rental_new_customer). Funziona grazie al reverse-lookup nel resolver, ma e' fragile: se in futuro il reverse-lookup viene rimosso o l'admin disabilita il canonical, questo unico invio diventa l'unico a saltare.

3.3 — supercar_day_before / utilitaria_day_before fuori dal sistema Pro

File: trigger-reminders.ts:106, send-booking-reminders.ts:246

Questi getTemplate(key) leggono direttamente dalla mappa locale messageTemplates[key], bypassando il resolver. Non rispettano handled_events, non rispettano target_service_type. L'admin che li modifica da MessaggiSistemaProTab li trova solo se la riga ha message_key='supercar_day_before' esatta (chiave NON pro_, anomala).

3.4 — CustomersTab.tsx: custom send bypassa wrapper + altre vars

File: CustomersTab.tsx:1394-1404

Invio promozione tier costruisce manualmente tpl.message_body.replace(/\{nome\}/g, firstName) e passa customMessage invece di templateKey. NON applica il wrapper header/footer anche se include_header=true sul template. Se l'admin scrive {cliente} o {full_name} nel template, restano letterali (solo {nome} e' sostituito).

4. Findings cleanup

4.1 — pro_dr7_privilege_noleggio / lavaggio senza voce in EVENT_DESCRIPTIONS

File: netlify/functions/utils/dr7Privilege.ts:107

L'UI Programmazione di MessaggiSistemaProTab non sa quando questi template partono perche' la mappa getProKeyEventTriggers non li copre. L'admin che modifica il template non capisce in che condizioni viene inviato.

4.2 — include_header ignorato in 2 sender

File:

  • send-checkin-checkout-whatsapp.ts:196

Wrappano sempre header/footer anche se il template ha include_header=false. Inconsistente con il resto del sistema.

8. Interventi suggeriti, ordine consigliato

  1. Aggiungere 12 chiavi mancanti a OLD_TO_PRO in src/utils/proTemplateRouting.ts e creare i record corrispondenti in system_messages (Messaggi di Sistema Pro). Definire anche EVENT_DESCRIPTIONS per ognuna cosi' la UI Programmazione le mostra.
  2. Allineare nexi-nuovo-addebito.ts:63 a passare anche contract_number (non solo contract_ref), nome, importo per parita' con gli altri caller di pro_email_addebito.
  3. Risolvere nexi-payment-callback.ts:1086: passare notifyAdmin: true (o customPhone admin esplicito) cosi' il guard no_recipient_phone non blocca prima del resolver.
  4. Estendere vars pro_email_contratto (send-contract-email.ts:102) con almeno pickup_date, dropoff_date, plate, contract_number, total.
  5. Safety-net "strip remaining {var}": oggi il cleanup degli placeholder non sostituiti vive solo in send-whatsapp-notification.ts:290. Estenderlo a tutti i sender che bypassano (in particolare email).
  6. Fallback per le 5 label hardcoded: ricerca prima per message_key canonical (es. pro_promozione_elite), poi per label come backup.
  7. Migrare supercar_day_before / utilitaria_day_before a pro_promemoria_estensione_* per uniformita' con il resto del sistema (rispettare handled_events, target_service_type).
